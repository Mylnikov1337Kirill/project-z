# Вопросы и ответы Project Z

Файл для редакторского просмотра: тексты вопросов и вариантов ответов сгруппированы по главам.

## Глава 1. ИИ как инженерный инструмент

### Сцена 1.1: Кто автор изменений?

Вопрос

ИИ-агент подготовил пул-реквест: тесты зелёные, но в расчёте цены есть изменение, которое автор не может объяснить. Что делать перед ревью?

Ответы

- Неверный: Влить изменения: проверки же прошли.
- Неверный: Попросить ревьюера разобраться и решить за автора.
- Верный: Остановиться: понять логику, попросить ИИ объяснить спорный фрагмент и убрать непонятное.
- Неверный: Оставить спорный расчёт, но не упоминать его в описании: если тесты зелёные, ревью само поймает проблему.
- Неверный: Попросить ИИ добавить ещё тестов на расчёт цены, не разбирая саму спорную логику.

### Сцена 1.2: Готово ли это к ревью?

Вопрос

Выбери признаки изменений с ИИ-помощью, которые действительно готовы к обычному ревью.

Ответы

- Верный: Решают одну конкретную задачу
- Неверный: Изменений много, значит работа солидная
- Верный: Не трогают соседние модули без запроса
- Неверный: ИИ уверенно сообщил, что всё правильно
- Верный: Есть понятный критерий приёмки
- Неверный: Заодно переписан соседний модуль
- Верный: Автор может объяснить каждое существенное изменение
- Неверный: Ревьюер сам восстановит контекст
- Верный: Есть проверка или честно указано, почему её нет
- Неверный: Пул-реквест выглядит сложным и умным

### Сцена 1.3: Где риск?

Вопрос

Цель: добавить предупреждение на форму отпуска. В пул-реквесте изменён компонент формы и добавлен текст предупреждения, но ещё переписана проверка дат, переименованы поля в контракте данных, обновлены снимки интерфейса; отдельной проверки предупреждения нет. Отметь риски перед ревью.

Ответы

- Верный: Пул-реквест вышел за границы задачи
- Неверный: В задаче прямо названа форма отпуска
- Верный: Проверка дат переписана без запроса
- Неверный: Компонент формы изменён, чтобы показать предупреждение
- Верный: Поля в контракте данных переименованы без объяснения
- Неверный: Добавлен текст предупреждения
- Верный: Снимки не доказывают поведение предупреждения
- Неверный: Снимки обновлены вместе с изменением интерфейса
- Верный: Нет проверки видимого поведения предупреждения
- Неверный: Снимки обновлены, значит проверка не нужна

### Сцена 1.4: Собери самопроверку перед ревью

Вопрос

Собери этапы самопроверки перед ревью: от рамки задачи и понимания diff к очистке, безопасности, доказательствам и заметке.

Ответы

- Рамка задачи: сверить набор изменений с заявленной задачей
- Граница данных: проверить, что секреты, персональные данные и дампы не уходили в модель
- Доказательства: запустить проверки или честно указать, почему их нет
- Авторское понимание: объяснить каждое существенное изменение
- Передача на ревью: отметить ИИ-помощь по командному процессу
- Чистка diff: убрать лишнее форматирование, переименования и соседний рефакторинг

Правильный порядок:
1. Рамка задачи: сверить набор изменений с заявленной задачей
2. Авторское понимание: объяснить каждое существенное изменение
3. Чистка diff: убрать лишнее форматирование, переименования и соседний рефакторинг
4. Граница данных: проверить, что секреты, персональные данные и дампы не уходили в модель
5. Доказательства: запустить проверки или честно указать, почему их нет
6. Передача на ревью: отметить ИИ-помощь по командному процессу

### Финал главы 1: Финальный бой за пул-реквест

Вступление к финалу

ИИ-агент подводит неоднозначный набор изменений к ревью-шлюзу. Не ищи плохие слова по диагонали: восстанови задачу, отдели нормальные изменения от риска и реши, что можно принять на себя.

#### Раунд 1: сканер риска

Вопрос

Задача: перед удалением проекта показать предупреждение, если у проекта есть активные деплои. Агент принёс набор изменений из 9 файлов. В нём есть новая проверка `hasActiveDeployments`, текст в модалке, тесты, обновлённые снимки и несколько изменений вокруг прав доступа. Отметь только те пункты, которые должны остановить автора до открытия пул-реквеста.

Ответы

- Верный: `canDeleteProject` меняет поведение для админов, автор не может объяснить зачем
- Неверный: `hasActiveDeployments` добавлен к project helpers и покрыт unit-тестом
- Верный: Новый общий guard меняет тексты ещё в двух соседних модалках
- Неверный: Целевая модалка показывает предупреждение с названием проекта
- Верный: Тест кликает “Удалить”, но не создаёт проект с активным деплоем
- Неверный: Снимок обновлён из-за нового текста предупреждения

#### Раунд 2: ход автора

Вопрос

После первого прохода CI зелёный, но остались два спорных места: автор не понимает изменение `canDeleteProject`, а общий `useDangerousActionGuard` трогает соседние модалки. При этом продукт просит показать предупреждение сегодня. Какой следующий ход удерживает ответственность за набор изменений?

Ответы

- Неверный: Открыть всё как есть, но честно написать ревьюеру: “особенно посмотри права и общий хук”.
- Неверный: Оставить общий хук и смену прав, но добавить TODO: вернуться к ним после релиза предупреждения.
- Неверный: Отменить все изменения и написать руками: раз агент ошибся в соседних файлах, использовать его результат нельзя.
- Неверный: Попросить агента написать убедительное объяснение для `canDeleteProject` и открыть ревью вместе с ним.
- Верный: Сузить пул-реквест до модалки и `hasActiveDeployments`, убрать общий хук, а изменение прав разобрать отдельно до ревью.

#### Раунд 3: заряд доказательств

Вопрос

Автор сузил изменения: остались модалка удаления, `hasActiveDeployments` и тесты. Какие записи в самопроверке реально помогают ревьюеру понять, что результат готов к обычному ревью?

Ответы

- Верный: Границы: пул-реквест меняет только удаление проекта и проверку активных деплоев
- Неверный: Снимки интерфейса обновились, поэтому отдельное описание поведения не нужно
- Верный: Проверка покрывает сценарии “есть активный деплой” и “деплоев нет”
- Верный: Доменное правило сверено с существующим `ProjectDeploymentStatus`
- Неверный: В отчёте агента написано, что он удалил все лишние изменения
- Верный: В заметке к ревью указано, что ИИ помогал с тестом и стоит проверить условие показа

#### Раунд 4: ревью-шлюз

Вопрос

Перед финальным шлюзом остаётся маленький набор изменений: предупреждение показывается только при активных деплоях, права доступа вернулись к прежнему helper, тесты покрывают оба состояния, а в описании пул-реквеста есть отметка об ИИ-помощи и просьба проверить доменное условие. Но агент ещё предлагает “для чистоты” оставить общий `useDangerousActionGuard` в этом же пул-реквесте. Что делать?

Ответы

- Неверный: Открыть пул-реквест вместе с `useDangerousActionGuard`: абстракция пригодится в будущем, а тесты зелёные.
- Верный: Открыть пул-реквест без общего хука: оставить маленькие изменения, проверки и заметку для ревьюера.
- Неверный: Открыть с общим хуком, но в описании сфокусироваться только на предупреждении, чтобы ревью было быстрее.
- Неверный: Не открывать пул-реквест: если ИИ предлагал лишнюю абстракцию, весь результат надо переписать вручную.
- Неверный: Открыть всё и попросить ревьюера решить, стоит ли выносить общий хук отдельно.

## Глава 2. Постановка задачи

### Сцена 2.1: Поймать результат

Вопрос

Команда приносит агенту просьбу: “Нужно привести страницу заказа в порядок, пользователи путаются с оплатой”. Какая цель лучше превращает это в задачу?

Ответы

- Неверный: Сделать страницу заказа красивее и понятнее, чтобы пользователи меньше ошибались.
- Неверный: Улучшить страницу так, чтобы конверсия оплаты выросла и обращений в поддержку стало меньше.
- Неверный: Изучить страницу заказа, найти все проблемы и поправить всё, что мешает оплате.
- Верный: На странице заказа пользователь видит предупреждение, если платёж отклонён, и кнопку “Повторить оплату”.
- Неверный: Пересобрать весь checkout-флоу вокруг более понятной оплаты и нового состояния ошибки.

### Сцена 2.2: Очертить границы

Вопрос

Задача: добавить предупреждение в форму отпуска, если дата окончания раньше даты начала. Выбери строки, которые хорошо задают границы работы и запреты для агента.

Ответы

- Неверный: Заодно привести к одному виду все формы отпусков
- Верный: Можно читать форму отпуска, date helper и существующие тесты формы
- Неверный: Переименовать `startDate` и `endDate`, если так будет чище
- Верный: Можно менять компонент формы и один spec-файл для этой проверки
- Неверный: Переписать date helper, чтобы проверка стала универсальной
- Верный: Не менять API-контракт и имена полей дат
- Неверный: Можно править любые файлы, если агент сочтёт это нужным
- Верный: Не переписывать общий date picker

### Сцена 2.3: Выбрать примеры

Вопрос

Нужно попросить агента добавить пустое состояние в список проектов. Какие материалы лучше дать как примеры, чтобы помочь стилю и не сжечь контекст?

Ответы

- Неверный: Весь каталог `src`, чтобы агент сам нашёл похожее
- Верный: `TeamList.tsx`: соседнее пустое состояние в таком же стиле
- Неверный: Десять скриншотов разных страниц без пояснений
- Верный: `ProjectList.test.tsx`: существующий тест на пустой список
- Неверный: Ссылка на старую ветку, где похожий интерфейс уже удалили
- Верный: Текст из задачи: “Пока нет проектов. Создайте первый проект”
- Неверный: Вся дизайн-документация продукта на 40 страниц

### Сцена 2.4: Собрать бриф

Вопрос

Собери этапы брифа задачи так, чтобы агент сначала увидел результат и рамку работы, а потом критерии готовности и проверку.

Ответы

- Рабочая зона: какие файлы и модули можно читать или менять
- Ориентиры: 1-3 близких примера
- Результат: наблюдаемая цель для пользователя или системы
- Запреты: что нельзя менять даже “заодно”
- Проверка: команды и ручной сценарий
- Готовность: критерии приёмки задачи

Правильный порядок:
1. Результат: наблюдаемая цель для пользователя или системы
2. Рабочая зона: какие файлы и модули можно читать или менять
3. Запреты: что нельзя менять даже “заодно”
4. Ориентиры: 1-3 близких примера
5. Готовность: критерии приёмки задачи
6. Проверка: команды и ручной сценарий

### Финал главы 2: Финальный бой за бриф задачи

Вступление к финалу

Перед агентом открывается мутная задача. Пройди четыре шлюза: найди провалы в постановке, собери компактный бриф, выбери критерии и реши, когда нужен план до изменений.

#### Раунд 1: сканер мутной задачи

Вопрос

Заявка звучит так: “Добавь тесты на форму отпуска, чтобы было нормально”. Какие проблемы надо исправить до передачи задачи агенту?

Ответы

- Верный: Нет наблюдаемой цели: какое поведение формы должно проверяться
- Верный: Не указаны границы: какие файлы можно читать и менять
- Неверный: Заявка плохая только потому, что в ней вообще упомянуты тесты
- Верный: Нет 1-3 примеров тестов в нужном стиле
- Верный: Нет критериев приёмки: когда задача считается закрытой
- Неверный: Заявку обязательно надо переписать на английский
- Верный: Нет конкретной команды или ручного сценария проверки

#### Раунд 2: выбрать контракт

Вопрос

Нужно проверить ошибку в форме отпуска: дата окончания не может быть раньше даты начала. Какой вариант лучше готов к передаче агенту?

Ответы

- Верный: Цель: тест покрывает ошибку, когда дата окончания раньше даты начала. Менять только spec формы. Не трогать рабочий код и тестовые данные. Пример: `vacation-create.spec.ts`. Критерий: тест падает без проверки дат. Проверка: `npm run test:e2e -- vacation-create.spec.ts`.
- Неверный: Добавь тесты на форму отпуска и поправь всё, что мешает корректной работе дат.
- Неверный: Проверь форму отпуска, поправь рабочий код при необходимости и добавь тесты там, где увидишь пробелы.
- Неверный: Сначала составь план полного покрытия модуля отпусков, затем добавь все недостающие тесты и улучши локаторы.
- Неверный: Добавь проверку текста ошибки для любых некорректных дат, пример и команду запуска можно выбрать самому.

#### Раунд 3: заряд приёмки

Вопрос

Бриф просит добавить предупреждение в модалку удаления проекта, если есть активные деплои. Какие строки подходят для критериев приёмки и проверки?

Ответы

- Верный: Если есть активный деплой, модалка показывает предупреждение
- Неверный: Модалка выглядит аккуратнее и современнее
- Верный: Если активных деплоев нет, существующий сценарий удаления не меняется
- Верный: `npm run test -- ProjectDeleteDialog.test.tsx`
- Неверный: Агент сообщил, что всё работает
- Верный: `npm run lint`
- Неверный: Снимки интерфейса обновлены, значит поведение подтверждено
- Верный: Manual: открыть проект с активным деплоем и проверить текст предупреждения

#### Раунд 4: плановый шлюз

Вопрос

Задача затрагивает биллинг API, миграцию и несколько экранов. Бриф уже содержит цель, границы, запреты, примеры, критерии приёмки и проверку. Что лучше сделать перед изменениями?

Ответы

- Неверный: Раз бриф полный, сразу разрешить изменения во всех упомянутых модулях.
- Неверный: Попросить сначала сделать первый патч, а план и риски описать уже после него.
- Верный: Попросить агента изучить релевантные файлы без изменений и вернуть понимание задачи, список файлов, риски, план маленьких изменений и проверки.
- Неверный: Убрать границы из брифа, чтобы агент сам выбрал архитектурно правильный путь.
- Неверный: Разделить задачу на несколько веток и запустить агента в каждой, чтобы быстрее увидеть варианты решения.

## Глава 3. Работа от плана

### Сцена 3.1: Когда нужен план?

Вопрос

К агенту приходит задача: исправить расчёт скидки в старом модуле заказов, добавить регрессионный тест и не сломать отчётность. Какой первый ход удерживает контроль?

Ответы

- Неверный: Попросить сразу предложить новую схему скидок для всего модуля, чтобы не чинить баг точечно.
- Неверный: Сразу попросить исправить расчёт: агент сам поймёт, какие файлы менять.
- Неверный: Отнестись к задаче как к обычной автоподстановке: принять первый короткий патч и проверить потом.
- Неверный: Попросить агента добавить только тест, а расчёт скидки оставить на следующий проход.
- Верный: Попросить изучить релевантные файлы без изменений и вернуть план, риски, маленький набор изменений и проверки.

### Сцена 3.2: Прочитать план агента

Вопрос

Агент вернул план перед изменениями. Отметь признаки плана, которому можно дать шанс на маленький набор изменений.

Ответы

- Верный: Коротко повторяет наблюдаемую цель задачи
- Неверный: Обещает полностью улучшить модуль за один проход
- Верный: Отделяет файлы для чтения от файлов, которые планирует менять
- Неверный: Пишет общий подход, но не называет конкретные файлы
- Верный: Называет риски: контракт, миграции, соседние сценарии или слабые тесты
- Верный: Предлагает один маленький проверяемый набор изменений
- Неверный: Сразу прикладывает изменения, хотя просили пока не менять код
- Верный: Называет команды и ручной сценарий проверки

### Сцена 3.3: Согласовать границы

Вопрос

План агента в целом полезный, но он предлагает вместе с фиксами переписать общий helper, переименовать DTO и обновить все тесты модуля. Какие решения помогают удержать границы?

Ответы

- Верный: Разрешить только фикс расчёта и один регрессионный тест
- Неверный: Разрешить весь план: раз агент увидел helper, пусть сразу наведёт порядок
- Верный: Явно запретить переименование DTO в этом изменении
- Верный: Вынести переписывание общего helper в отдельную задачу
- Неверный: Убрать ограничения, чтобы не мешать архитектурно правильному решению
- Верный: Дать близкий пример существующего теста, если стиль неочевиден
- Неверный: Оставить переименование DTO ревьюеру: он решит, вредно это или нет
- Верный: Поставить условие остановки: если нужен второй модуль, вернуться с новым планом

## Глава 6. Дисциплина проверки

### Сцена 6.1: Доказать, а не поверить

Вопрос

Ответы

- Верный: Запустить релевантный тест, lint/build по необходимости, проверить интерфейс или доменный источник и записать оставшийся риск.
- Неверный: Попросить агента ещё раз пройти логику мысленно и перечислить, почему она должна работать.
- Неверный: Открыть ревью: агент уже объяснил, почему тесты должны пройти.
- Неверный: Попросить ревьюера самому проверить бизнес-правило, если код выглядит чистым.
- Неверный: Не запускать проверку: изменение выглядит маленьким, а тесты всё равно проверит CI после ревью.

### Сцена 6.2: Собрать матрицу проверки

Вопрос

Ответы

- Верный: Изменение интерфейса -> сборка + ручная проверка/снимок -> риск адаптивных состояний -> ревьюер смотрит экранное поведение
- Неверный: Изменение интерфейса -> только сборка -> риска нет -> ревьюер смотрит форматирование
- Верный: Исправление бага -> воспроизведение до фикса + тест после -> риск крайнего случая -> ревьюер смотрит падение без фикса
- Неверный: Бизнес-логика -> агент сам перечитал изменения -> риск закрыт -> ревьюер может пропустить правило
- Верный: Mapping/DTO -> тест на поля и nullability -> риск совместимости -> ревьюер смотрит контракт данных
- Неверный: Рефакторинг -> один снимок -> риск поведения закрыт -> ревьюер смотрит только интерфейс
- Верный: Бизнес-логика -> регрессионный тест + доменный источник -> риск неверного правила -> ревьюер смотрит правило и крайние случаи

Вопрос

- Верный: Проходит путь через форму так, как это делает пользователь
- Неверный: Завязался на длинную цепочку CSS-классов компонента
- Верный: Ищет элементы по роли, подписи или понятному тестовому идентификатору

- Неверный: Читает внутреннее состояние формы напрямую
- Верный: Падает, если убрать проверяемое бизнес-правило
- Неверный: Подменяет весь сценарий заглушками так, что интеграционный риск исчезает из теста
